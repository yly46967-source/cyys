## 文档说明

本文档定义了"悬赏接单托管交付平台"（以下简称"平台"）的产品需求。平台的核心定位是**托管交付平台**，而非简单的信息撮合平台，重点解决软件开发过程中的信任、透明度和可控性问题。

### 核心价值主张

- 把"选人"转化为"选结果"
- 提供需求澄清、验收标准、过程透明、风险兜底、售后维护的全流程保障
- 用制度和工具替代熟人信任，降低交易和管理成本

## 1. 产品定位与目标

### 1.1 产品定位

**托管交付平台** - 提供从需求澄清到交付验收的全流程托管服务

核心差异点：
- ❌ **不是**：信息撮合平台（只提供联系方式，线下成交）
- ✅ **而是**：托管交付平台（需求澄清→方案拆解→过程管理→验收测试→交付物归档→售后期）

### 1.2 目标用户

| 用户类型 | 核心特征 | 核心痛点 |
|---------|---------|---------|
| **客户** | 有明确业务目标但内部研发不足的创业公司、中小企业 | 找不到靠谱开发者、需求说不清、担心烂尾、不懂技术无法验收 |
| **开发者** | 自由职业开发者、小型开发团队 | 需求不清晰、付款不安全、客户要求不明确、争议难处理 |
| **平台运营** | 平台运营人员 | 供给质量参差不齐、交易过程不透明、争议处理困难 |

### 1.3 核心业务目标

**短期目标（MVP阶段）**：
- 验证"托管交付"模式的市场需求
- 完成最小闭环：需求发布→报价→里程碑托管→验收
- 积累首批可信交付案例

**中期目标**：
- 建立完善的供给质量体系
- 形成标准化的需求澄清和验收流程
- 实现争议处理的自动化和可预期化

**长期目标**：
- 成为垂直场景（如AI自动化、内部工具）的首选托管交付平台
- 建立开发者交付画像体系
- 实现长期服务包的持续收入

## 2. 用户角色定义

### 2.1 角色：客户（Client）

**角色描述**：有软件开发需求但缺乏技术能力或开发资源的个人或企业

**核心诉求**：
- 需求能够被理解和准确报价
- 能够判断开发者的靠谱程度
- 交付过程透明可见
- 验收标准清晰明确
- 资金安全，烂尾有保障

**关键行为**：
- 发布需求（选择模板或自定义描述）
- 选择是否需要需求澄清服务
- 查看和筛选报价
- 选择开发者并确认里程碑
- 验收里程碑并释放资金
- 提交质保期内缺陷修复请求

### 2.2 角色：开发者（Developer）

**角色描述**：提供软件开发服务的个人或团队

**核心诉求**：
- 需求清晰明确，避免反复扯皮
- 报价有依据，不被恶意压价
- 付款安全，不担心拖欠
- 争议有依据，不被主观评价
- 能够展示真实的交付能力

**关键行为**：
- 实名认证和能力验证
- 浏览需求并提交报价
- 接受项目后拆分里程碑
- 开发并提交交付物（代码、文档、可运行版本）
- 响应客户反馈和缺陷修复

### 2.3 角色：平台运营（Admin）

**角色描述**：负责平台运营、审核、争议处理的工作人员

**核心诉求**：
- 供给质量可控
- 交易过程透明
- 争议处理有据可依
- 数据可追踪和分析

**关键行为**：
- 审核开发者准入
- 审核复杂项目的需求文档
- 处理争议仲裁
- 监控交易风险

## 3. 核心业务流程

### 3.1 主流程：需求发布到交付验收

```
┌─────────────────────────────────────────────────────────────────────┐
│                          客户侧流程                                  │
└─────────────────────────────────────────────────────────────────────┘
   ① 登录/注册
   ② 发布需求
      - 选择需求类型（模板化/自定义）
      - 填写需求信息
      - 系统评估（价格区间、交付模式推荐）
   ③ 选择是否需要需求澄清（Discovery）
      - 如需要，支付Discovery费用
      - 平台需求顾问介入，输出结构化需求文档
   ④ 确认发布需求
   ⑤ 收到报价通知
   ⑥ 查看报价（智能降噪筛选）
      - 查看风险分析
      - 查看方案对比
   ⑦ 选择开发者
   ⑧ 确认里程碑和验收标准（双签）
   ⑨ 预存全款到托管账户
   ⑩ 监控交付进度
       - 查看里程碑时间线
       - 查看证据链（代码、文档、Demo）
       - 查看验收清单通过率
   ⑪ 收到验收申请
   ⑫ 验收里程碑
       - 勾选验收清单
       - 提交问题或确认通过
   ⑬ 确认后资金自动释放给开发者
   ⑭ 重复⑩-⑬直到所有里程碑完成
   ⑮ 进入质保期（默认30天）
   ⑯ 提交缺陷修复请求（如有）

┌─────────────────────────────────────────────────────────────────────┐
│                          开发者侧流程                                │
└─────────────────────────────────────────────────────────────────────┘
   ① 注册/登录
   ② 实名认证
   ③ 能力验证（可选）
      - 技术挑战测试
      - 过往项目核验
   ④ 等待审核通过
   ⑤ 浏览需求市场
      - 按场景/技术栈/交付模式筛选
   ⑥ 查看需求详情
      - 结构化需求文档
      - 客户背景信息
   ⑦ 提交报价
      - 选择交付模式（里程碑/小时/混合）
      - 填写报价和工期
      - 里程碑拆分建议
      - 上传方案文档（可选）
   ⑧ 等待客户选择
   ⑨ 收到项目通知
   ⑩ 确认里程碑和验收标准（双签）
   ⑪ 开始交付
       - 拆分任务
       - 开发并提交代码
       - 上传交付物
   ⑫ 提交验收申请
   ⑬ 等待客户验收
   ⑭ 处理客户反馈（如有问题）
   ⑮ 收到资金释放
   ⑯ 进入下一里程碑
   ⑰ 质保期内响应缺陷修复请求
```

### 3.2 子流程A：需求澄清（Discovery）

**触发条件**：
- 客户发布需求时选择"需要需求澄清"
- 系统评估认为需求不确定性高，强制推荐Discovery

**流程步骤**：
```
① 客户选择Discovery并支付费用
② 系统分配需求顾问
③ 需求顾问与客户沟通（站内聊天/视频）
④ 需求顾问输出结构化需求文档，包括：
   - 需求背景和目标
   - 功能列表和优先级
   - 验收标准（可测试条件）
   - 技术方案建议
   - 里程碑拆分建议
   - 风险提示
⑤ 客户确认需求文档
⑥ 需求文档更新到需求详情页
⑦ 开发者基于结构化文档报价
```

### 3.3 子流程B：争议处理

**触发条件**：
- 客户拒绝验收但开发者认为已完成
- 开发者认为客户需求变更但未走流程
- 其他交付过程中的分歧

**流程步骤**：
```
① 一方发起争议（选择争议类型 + 提交证据）
② 对方收到通知，提交陈述和证据
③ 双方进入协商期（3天）
   - 可选择继续协商
   - 可选择请求平台介入
④ 如协商成功，关闭争议
⑤ 如协商失败，平台介入仲裁
   - 仲裁员查看证据链
   - 基于验收标准和合同裁决
   - 资金重新分配
⑥ 仲裁结果归档（影响双方信誉）
```

### 3.4 子流程C：证据链归档

**触发时机**：每个里程碑提交验收时

**归档内容**：
```
① 代码提交记录
   - Git仓库链接（平台托管）
   - 提交哈希和说明
   - 代码审查记录（如有）
② 交付物
   - 可运行版本（部署地址/下载链接）
   - 关键文档（需求文档、设计稿、测试用例、部署手册）
③ 验收清单
   - 客户勾选记录
   - 未通过项的说明
④ 沟通记录
   - 关键决策记录
   - 变更单记录
```

## 4. 功能模块详细说明

### 4.1 模块一：用户认证与权限管理

#### 4.1.1 注册与登录

**功能描述**：支持用户注册和登录

**页面路径**：`/auth/register`, `/auth/login`

**核心字段**：
- 手机号/邮箱
- 密码
- 验证码
- 用户角色选择（客户/开发者）

**业务规则**：
- 手机号/邮箱唯一性校验
- 密码强度要求（至少8位，包含字母和数字）
- 未登录用户只能浏览需求，不能发布和抢单

#### 4.1.2 实名认证（开发者必需）

**功能描述**：开发者必须完成实名认证才能接单

**页面路径**：`/developer/verification`

**核心字段**：
- 真实姓名
- 身份证号
- 身份证照片（正反面）
- 手持身份证照片
- 联系方式
- 银行账户信息

**业务规则**：
- 实名信息加密存储
- 平台运营审核通过后才能接单
- 一个身份证只能注册一个开发者账号（反多号）

#### 4.1.3 能力验证（可选但推荐）

**功能描述**：通过技术测试和项目核验提升开发者可信度

**页面路径**：`/developer/skills`

**验证方式**：
- 标准化技术挑战（在线编码测试）
- 过往项目核验（可验证的项目链接）
- 试单评估（小额带验收的真实项目）

**输出**：
- 开发者能力标签（技术栈、场景经验）
- 交付能力评分（待项目完成后生成）

### 4.2 模块二：需求管理（客户端）

#### 4.2.1 需求发布

**功能描述**：客户发布软件开发需求

**页面路径**：`/client/requirements/create`

**需求类型选择**：
1. **模板化需求**（适合标准化项目）
   - 官网/落地页
   - 标准CRUD内部工具
   - 常见第三方集成
   - 数据看板
   - AI自动化脚本

2. **自定义需求**（适合复杂项目）

**核心字段**：
- 需求标题
- 需求类型（模板/自定义）
- 需求描述（富文本编辑器）
- 预算范围
- 期望交付时间
- 技术栈要求（可选）
- 参考资料上传（设计稿、文档、链接等）

**智能评估**：
- 系统分析需求复杂度
- 推荐交付模式（里程碑/小时/混合）
- 提供价格区间预估（基于历史数据）
- 评估是否需要Discovery

**交互流程**：
```
选择需求类型 → 填写模板/自定义描述 → 上传参考资料 → 
系统评估 → 选择是否需要Discovery → 确认发布
```

#### 4.2.2 需求澄清（Discovery）

**功能描述**：需求顾问帮助客户把模糊需求结构化

**页面路径**：`/client/requirements/:id/discovery`

**服务流程**：
- 客户支付Discovery费用（固定价格，如¥500-2000）
- 系统分配需求顾问
- 顾问与客户沟通（站内聊天/预约视频）
- 顾问输出结构化需求文档
- 客户确认后，需求文档更新到需求详情页

**输出文档结构**：
```markdown
# 需求澄清文档

## 1. 需求背景
- 业务目标
- 目标用户
- 核心痛点

## 2. 功能列表
- 核心功能（必须有）
- 重要功能（应该有）
- 边缘功能（可以有）

## 3. 验收标准
- 功能验收标准（可测试条件）
- 性能验收标准
- 兼容性验收标准

## 4. 技术方案建议
- 推荐技术栈
- 架构设计建议
- 第三方服务集成建议

## 5. 里程碑拆分建议
- 里程碑1：XXX（XX天，XX元）
- 里程碑2：XXX（XX天，XX元）
...

## 6. 风险提示
- 技术风险
- 进度风险
- 成本风险
```

#### 4.2.3 需求管理

**功能描述**：客户查看和管理自己的需求

**页面路径**：`/client/requirements`

**功能点**：
- 需求列表（全部/进行中/已完成）
- 需求状态查看
- 编辑需求（未收到报价前）
- 关闭需求

### 4.3 模块三：报价与匹配

#### 4.3.1 需求市场（开发者侧）

**功能描述**：开发者浏览和筛选需求

**页面路径**：`/developer/marketplace`

**筛选维度**：
- 按场景筛选（电商、内部工具、AI自动化等）
- 按技术栈筛选（React、Vue、Python等）
- 按交付模式筛选（里程碑/小时/混合）
- 按预算范围筛选
- 按发布时间筛选

**需求列表展示**：
- 需求标题
- 需求类型标签
- 预算范围
- 期望交付时间
- 报价数量
- 需求状态

**需求详情**：
- 完整需求描述
- 结构化需求文档（如有Discovery）
- 客户基本信息
- 历史合作记录（如有）

#### 4.3.2 报价提交

**功能描述**：开发者提交报价和方案

**页面路径**：`/developer/requirements/:id/proposal`

**核心字段**：
- 交付模式选择（里程碑/小时/混合）
- 总报价
- 预估工期
- 里程碑拆分（里程碑模式）
  - 里程碑名称
  - 交付内容描述
  - 报价
  - 预估工期
- 小时费率（小时制）
- 方案文档（可选，支持上传PDF/链接）
- 相关案例展示

**智能报价助手**：
- 基于需求复杂度建议报价区间
- 里程碑拆分建议
- 提醒常见风险点

#### 4.3.3 报价筛选与对比（客户端）

**功能描述**：客户查看和筛选报价

**页面路径**：`/client/requirements/:id/proposals`

**智能降噪**：
- 排除高风险报价（开发者信誉低、报价异常）
- 按"同模式可比"分组（里程碑 vs 小时）
- 只展示3-5个推荐方案

**方案对比维度**：
- 报价合理性分析（与历史数据对比）
- 交付方式对比
- 开发者交付画像
  - 按期交付率
  - 验收一次通过率
  - 缺陷密度
  - 客户评价

**风险雷达**：
- 延期概率评估
- 交付质量评估
- 沟通响应评估

**交互流程**：
```
收到报价通知 → 查看智能筛选结果 → 
查看方案对比 → 查看开发者画像 → 
选择开发者 → 确认里程碑和验收标准
```

#### 4.3.4 开发者选择与里程碑确认

**功能描述**：客户选择开发者并确认里程碑

**页面路径**：`/client/requirements/:id/confirm`

**核心流程**：
- 选择开发者报价
- 查看里程碑拆分
- 确认/调整验收标准（双签）
- 确认合同条款
- 预存全款到托管账户

**验收标准模板**（根据需求类型自动生成）：
- 功能验收清单
- 性能验收指标
- 兼容性验收要求
- 文档验收要求

### 4.4 模块四：里程碑交付与验收

#### 4.4.1 里程碑工作台（开发者侧）

**功能描述**：开发者管理任务和提交交付物

**页面路径**：`/developer/projects/:id/workspace`

**功能点**：
- 里程碑列表和时间线
- 任务拆分和管理
- 代码仓库关联（Git集成）
- 工时记录（小时制项目）
- 交付物上传
- 验收清单自检

**任务管理**：
- 创建任务
- 分配到里程碑
- 设置任务状态（待办/进行中/已完成）
- 任务描述和附件

**代码仓库集成**：
- 支持GitHub/GitLab/Gitee
- 自动同步提交记录
- 生成代码提交报告

**交付物提交**：
- 代码提交记录（自动同步）
- 可运行版本上传（部署地址/下载链接）
- 关键文档上传（需求文档、设计稿、测试用例、部署手册）
- 验收清单自检

#### 4.4.2 项目监控面板（客户端）

**功能描述**：客户查看项目进度和风险

**页面路径**：`/client/projects/:id/dashboard`

**可视化进度追踪**：
- 里程碑时间线（完成状态、延期风险）
- 验收清单通过率
- 每周Demo视频/可运行版本
- 风险雷达（新增需求数、未决问题数、延期概率）

**证据链查看**：
- 代码仓库访问（只读）
- 文档归档查看
- 可运行版本下载/访问

**里程碑详情**：
- 交付内容描述
- 验收标准
- 交付物列表
- 验收历史

#### 4.4.3 里程碑验收

**功能描述**：客户验收里程碑并释放资金

**页面路径**：`/client/projects/:id/milestones/:milestoneId/acceptance`

**验收流程**：
```

勾选验收清单 → 提交问题或确认通过 → 
资金释放（如有问题，开发者修复后重新提交）
```

**验收清单**：
- 功能验收项（可勾选）
- 性能验收项（可勾选）
- 文档验收项（可勾选）
- 其他自定义验收项

**问题提交**：
- 选择未通过的验收项
- 描述问题
- 上传截图/录屏
- 设置优先级

**资金释放规则**：
- 客户确认通过后，资金自动释放给开发者
- 如有争议，进入争议处理流程

## 8. 关键规则与约束

### 8.1 业务规则

#### 8.1.1 需求发布规则
- 客户必须完成实名认证才能发布需求
- 需求发布后可编辑，但收到报价后不可编辑
- 需求发布后7天内未收到报价可关闭

#### 8.1.2 报价规则
- 开发者必须通过实名认证和能力验证才能报价
- 一个开发者对同一需求只能提交一次报价
- 报价提交后48小时内可撤回

#### 8.1.3 里程碑规则
- 每个项目至少包含2个里程碑
- 单个里程碑金额不超过总金额的50%
- 里程碑必须包含明确的验收标准

#### 8.1.4 验收规则
- 客户收到验收申请后7天内必须验收
- 超时未验收视为自动通过
- 验收不通过需明确指出未通过的验收项
- 开发者需在3天内修复问题并重新提交

#### 8.1.5 资金托管规则
- 客户选择开发者后需预存全款到托管账户
- 里程碑验收通过后自动释放对应资金
- 争议期间资金冻结
- 质保期结束后释放质保金（如有）

#### 8.1.6 争议规则
- 任何一方可发起争议
- 争议发起后进入3天协商期
- 协商不成可请求平台介入
- 平台仲裁结果为最终结果

#### 8.1.7 质保规则
- 默认质保期30天
- 质保期内免费修复缺陷
- 功能变更不属质保范围
- 开发者需在48小时内响应缺陷

### 8.2 系统约束

#### 8.2.1 技术约束
- 前端使用React 18 + TypeScript
- 后端使用Node.js + NestJS
- 数据库使用PostgreSQL
- 文件存储使用阿里云OSS

#### 8.2.2 业务约束
- 单个项目金额不超过50万元（MVP阶段）
- 单个里程碑金额不超过10万元
- 质保期不超过90天

#### 8.2.3 合规约束
- 实名信息需符合《网络安全法》要求
- 资金托管需符合《支付服务管理办法》
- 合同模板需符合《合同法》要求

## 10. 风险与挑战

### 10.1 产品风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 冷启动困难 | 高 | 人工精选供给，人工撮合首批交易 |
| 绕开平台 | 高 | 提供不可替代的撮合后价值 |
| 争议处理困难 | 中 | 建立清晰的验收标准和仲裁规则 |
| 供给质量参差不齐 | 高 | 严格的准入和能力验证 |

### 10.2 技术风险

|------|------|---------|
