# 平台型 AI 项目交付保障产品——后续修改说明

**文档版本**：v1.0  
**编写日期**：2026-06-23  
**适用范围**：Mercenary-Guild / Assurance OS 前端后续重构  
**文档目标**：将“项目交易平台”与“AI 项目第三方保障服务”整合为一个完整、可信、可操作的产品。

---

## 一、修改背景

当前项目先后出现了两个偏向不同的产品版本：

1. 原始版本偏向项目交易平台：
   - 客户发布项目；
   - 服务商浏览项目并提交申请；
   - 用户可以查看项目、预算、周期和竞标信息；
   - 具备客户与服务商两类用户中心。

2. 当前 Next.js 版本偏向企业服务宣传平台：
   - 重点介绍项目体检、供应商评估、PoC 验收和实施监理；
   - 服务价值表达较清晰；
   - 但缺少项目市场、下单、报价、项目协作等平台能力；
   - 页面内容偏静态，用户缺少持续使用产品的理由。

这两个方向不是互相排斥的。

后续产品应定位为：

> 面向企业 AI 项目的交易与交付保障平台。客户可以发布项目、购买标准服务、筛选供应商、管理交付过程；供应商可以响应项目、提交方案和交付证据；平台提供独立评估、过程监理、验收与风险控制。

产品不是单纯的“自由职业接单平台”，也不是单纯的“咨询公司官网”，而是：

> **项目交易入口 + 第三方交付保障 + 项目协作工作台。**

## 二、产品定位调整

### 2.1 核心价值

对客户：

- 可以直接发布 AI 项目或购买标准服务；
- 可以看到项目预算、状态、报价和交付进度；
- 不必仅凭销售承诺选择供应商；
- 可以购买平台的供应商筛选、过程监理和验收保障；
- 项目过程和交付证据可追溯。

对供应商：

- 可以浏览真实项目机会；
- 可以提交结构化方案和报价；
- 明确知道需求边界和验收条件；
- 通过真实履约数据建立可信能力画像；
- 降低需求不清、验收主观和付款争议。

对平台：

- 项目发布和供应商撮合形成交易入口；
- 项目体检、供应商评估、实施监理和验收形成服务收入；
- 项目过程数据形成长期产品资产；
- 从一次性项目逐步延伸至持续运营陪跑。

### 2.2 核心产品结构

```text
公开首页
  ├─ 项目市场
  ├─ 发布项目
  ├─ 标准保障服务
  ├─ 项目案例
  └─ 登录/注册

客户工作台
  ├─ 我的项目
  ├─ 收到的方案与报价
  ├─ 已购买服务
  ├─ 项目风险与里程碑
  ├─ 验收与证据
  └─ 消息与决策事项

供应商工作台
  ├─ 可响应项目
  ├─ 我的方案
  ├─ 进行中项目
  ├─ 交付物与整改
  ├─ 能力画像
  └─ 消息中心

平台保障工作台
  ├─ 项目体检
  ├─ 供应商评估
  ├─ PoC 验收设计
  ├─ 里程碑监理
  ├─ 风险台账
  └─ 验收结论
```

## 三、用户角色调整

### 3.1 企业客户

主要操作：

- 发布项目需求；
- 选择公开招募或定向邀请；
- 查看供应商方案和报价；
- 购买平台保障服务；
- 确认里程碑和验收标准；
- 查看交付进度、风险和实际投入；
- 对交付物进行验收；
- 提交整改或争议。

### 3.2 项目供应商

供应商不再使用过于个人化的“开发者接单”定位。

供应商可以是：

- 软件公司；
- AI 解决方案公司；
- 小型交付团队；
- 独立专家团队；
- 测试或数据服务机构。

主要操作：

- 维护企业和团队资料；
- 浏览可响应项目；
- 提交方案、报价、团队成员和工期；
- 披露真实投入人员；
- 上传里程碑交付物；
- 响应风险、问题和整改要求；
- 积累履约画像。

### 3.3 平台监理顾问

主要操作：

- 审查项目需求；
- 设计供应商评分规则；
- 核验供应商团队和案例；
- 审查里程碑交付物；
- 维护风险台账；
- 发布周报和监理意见；
- 组织验收与复测；
- 输出通过、有条件通过或不通过结论。

### 3.4 平台管理员

主要操作：

- 用户和供应商审核；
- 项目内容审核；
- 服务订单管理；
- 顾问分配；
- 争议处理；
- 案例和内容管理；
- 平台运营数据查看。

## 四、核心业务模式

产品提供两种项目入口。

### 4.1 入口一：发布项目

适合已经有明确需求、希望寻找供应商的客户。

```text
发布项目
→ 平台进行需求完整度检查
→ 客户选择保障等级
→ 项目进入市场
→ 供应商提交方案和报价
→ 客户查看对比
→ 选择供应商
→ 确认合同、里程碑和验收标准
→ 项目执行
→ 里程碑验收
→ 项目结束
```

### 4.2 入口二：购买标准服务

适合需求尚不清晰、已有供应商或项目已经出现风险的客户。

服务产品包括：

1. AI 项目体检；
2. 供应商筛选与技术把关；
3. RFP 与比选支持；
4. PoC 验收设计；
5. 实施监理 Lite；
6. 全流程监理；
7. 验收评测；
8. 运营陪跑。

```text
选择服务
→ 填写项目情况
→ 查看服务范围、周期和价格
→ 提交服务订单
→ 平台分配顾问
→ 进入项目工作台
→ 按服务范围交付报告、评分、周报或验收结论
```

### 4.3 两种入口的融合

发布项目时，客户可选择：

- 仅发布项目；
- 发布项目 + 供应商评估；
- 发布项目 + PoC 验收设计；
- 发布项目 + 全流程监理。

购买标准服务后，也可以继续：

- 将体检结果转为正式项目；
- 基于 RFP 结果公开招募供应商；
- 将供应商评估结果转入实施监理；
- 将验收结果转入运营陪跑。

## 五、保障等级设计

为了使平台商业模式清晰，项目发布时增加保障等级。

### 基础发布

- 发布项目；
- 接收方案和报价；
- 基础消息通知；
- 客户自行选择和管理供应商。

### 优选保障

- 平台检查需求完整度；
- 供应商身份和案例基础核验；
- 方案结构化对比；
- 异常报价提示；
- 推荐 3 家候选供应商。

### 全程保障

- 项目体检；
- 供应商评估；
- PoC 指标设计；
- 里程碑监理；
- 风险台账；
- 交付证据归档；
- 验收与复测；
- 争议支持。

页面上不能只显示抽象的“推荐服务”，应明确展示：

- 包含什么；
- 不包含什么；
- 周期；
- 价格或询价方式；
- 适用项目；
- 最终交付物。

## 七、项目市场修改

恢复原“任务大厅”，但改名为：

> 项目市场

不再使用“任务”“抢单”“佣兵”等容易降低企业可信度的表达。

### 7.1 项目卡片

必须显示：

- 项目标题；
- 匿名客户行业；
- 项目类型；
- 项目阶段；
- 预算；
- 周期；
- 方案数量；
- 保障等级；
- 最近更新时间；
- 是否允许公开响应。

### 7.2 筛选条件

- 行业；
- 项目类型；
- 当前阶段；
- 预算范围；
- 预计周期；
- 保障等级；
- 是否已有供应商；
- 发布时间。

### 7.3 项目状态

```text
draft             草稿
reviewing         平台审核中
open              开放响应
evaluating        方案评估中
contracting       合同确认中
in_progress       执行中
accepting         验收中
warranty          质保/运营期
completed         已完成
paused            已暂停
cancelled         已取消
```

项目列表不得同时渲染“加载失败”和正常结果。

必须统一实现：

- Loading；
- Empty；
- Error；
- Loaded；
- Permission denied。

## 八、项目详情修改

项目详情不应只展示需求描述和接受任务按钮。

建议页面结构：

### 8.1 项目概览

- 项目背景；
- 业务目标；
- 当前阶段；
- 预算和周期；
- 保障等级；
- 是否已有供应商；
- 响应截止日期。

### 8.2 项目范围

- 必须完成；
- 可选范围；
- 明确不包含；
- 第三方系统；
- 数据范围；
- 交付环境。

### 8.3 验收要求

- 业务指标；
- 技术指标；
- 合规指标；
- 文档和知识转移要求。

### 8.4 平台风险提示

示例：

- 当前需求缺少明确测试集；
- 团队投入要求尚未量化；
- 外部系统接口责任边界不清；
- 项目建议购买 PoC 验收设计。

### 8.5 供应商响应

供应商提交：

- 方案摘要；
- 总报价；
- 周期；
- 里程碑；
- 实际投入成员；
- 相关案例；
- 风险假设；
- 不包含内容；
- 是否接受平台审计；
- 是否接受验收与版本留痕。

不能只提交一段“申请说明”。

## 十、匿名项目案例

所有对外页面隐藏真实企业名称、团队名称和敏感项目参数。

### 10.1 案例一：海事科技企业供应商重新遴选

对外标题：

> 某海事科技企业 AI 项目供应商重新遴选

行业：

> 港航与海事科技

背景描述：

> 客户此前推进 AI 项目时，原交付方在销售阶段对能力和范围作出过度承诺。项目进入收尾阶段后，数据治理、业务流程、系统集成、模型效果与验收标准等关键问题逐渐暴露，多项交付难以完成，项目无法顺利收尾。

平台介入目标：

- 重新梳理项目目标和边界；
- 识别前期失败原因；
- 建立新一轮供应商评分标准；
- 核验候选团队的实际交付经验；
- 审查技术方案是否具备落地条件；
- 将验收标准前置。

平台服务：

- 项目复盘与风险诊断；
- 需求和边界重构；
- 供应商筛选；
- 技术方案审查；
- 团队与案例核验；
- PoC 验收建议。

展示成果：

- 形成供应商评分矩阵；
- 明确关键技术和交付边界；
- 排除过度承诺但证据不足的候选方；
- 将选择依据从“销售表达”转为“真实交付证据”。

口语化页面文案：

> 很多 AI 项目不是想法不够，而是前期承诺太满。真正开始交付后，才发现数据、系统对接、模型效果和验收标准都没有讲清楚。我们帮助这家海事科技企业重新梳理边界、核验团队，并建立可落地的供应商选择标准。

### 10.2 案例二：智慧城市项目里程碑监理

对外标题：

> 某智慧城市科技企业 AI 项目履约监理

行业：

> 智慧城市与数字治理

背景描述：

> 客户的一项 AI 工程项目由外部供应商负责实施。供应商前期表示将投入完整交付团队，但实际项目中仅有极少人员承担主要工作，实际资源投入与承诺严重不符，导致交付数量不足、质量偏低、里程碑持续偏差。

平台介入目标：

- 持续核查实际投入人员；
- 审查每个里程碑交付物；
- 比对计划产出与实际产出；
- 跟踪问题整改；
- 在延期和质量风险扩大前及时预警。

平台服务：

- 项目资源投入监控；
- 里程碑交付审查；
- 周报与风险台账；
- 交付物完整性检查；
- 问题闭环；
- 阶段验收建议。

展示成果：

- 建立承诺投入与实际投入对比；
- 对交付缺口进行量化；
- 将质量问题关联到责任人和整改时间；
- 为客户提供可追责的项目证据链。

口语化页面文案：

> 客户听到的可能是一支完整团队在交付，但真正推进时，实际投入的人力、产出节奏和交付质量未必匹配承诺。我们帮助这家智慧城市企业持续核查资源投入，审查每个里程碑，并把问题、责任和整改要求记录下来。

### 10.3 案例隐私规则

禁止公开：

- 真实公司名称；
- 真实供应商名称；
- 真实人员姓名；
- 可识别项目名称；
- 合同金额；
- 内部系统地址；
- 未经授权的截图、合同和报告。

可以展示：

- 匿名行业；
- 风险类型；
- 服务方法；
- 脱敏后的流程；
- 经客户确认的成果摘要；
- 模拟或重新绘制的数据图表。

案例页面显著注明：

> 为保护客户隐私，案例中的企业名称、金额和部分项目细节已做匿名化处理。

## 十一、工作台业务逻辑修改

当前工作台只展示固定风险、供应商和指标，缺少角色行为。

### 11.1 客户工作台

增加：

## 十八、验收标准

修改后的版本必须满足：

### 内容

- 首页能够在十秒内说明“平台是什么”；
- 用户能明确找到“发布项目”和“浏览项目”；
- 服务能力不是孤立宣传，而是可以下单；
- 至少展示两个匿名真实案例；
- 页面不再混用 TechCraft、佣兵公会和 Assurance OS 等多套品牌概念。

### 业务

- 客户可以完成项目发布；
- 客户可以购买一种保障服务；
- 供应商可以查看项目并提交结构化方案；
- 客户可以看到方案数量和对比信息；
- 工作台能显示项目、订单、风险、里程碑和待处理事项；
- 演示数据刷新后可以恢复或重置。

### 交互

- 无死按钮；
- 无原生 `alert/confirm`；
- 无页面同时显示成功与错误状态；
- 所有异步操作有 Loading 和结果反馈；
- 表单具备校验、草稿和离开提醒；
- 桌面和移动端均无横向溢出。

### 隐私

- 案例不出现真实客户名称；
- 不出现真实供应商名称；
- 不出现可识别合同金额和人员；
- 案例页面包含匿名化声明。

---

## 十九、最终产品表达

产品对外不应再只说：

> 我们提供 AI 项目监理服务。

也不应只说：

> 这是一个开发者接单平台。

统一表达应为：

> 这是一个面向企业 AI 项目的交易与交付保障平台。企业可以在平台发布项目、选择供应商、购买第三方保障服务，并在统一工作台中管理里程碑、风险、交付证据与验收。

首页核心文案建议：

> **发布 AI 项目，找到真正能交付的团队。**  
> 从供应商选择到实施监理和验收，平台让每一次承诺都有证据，让每一个里程碑都可检查。
