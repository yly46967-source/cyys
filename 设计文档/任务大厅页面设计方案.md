# 任务大厅页面设计方案 v2.0

## 一、文档目标

本文档用于定义站点第二个页面“任务大厅”的页面结构、筛选交互、数据组织和实现约束，作为 `index.html` 与 `task-hall.html` 的设计与前端实现依据。

本方案基于当前首页的白底 Dribbble 风格延展，不引入另一套深色视觉体系，避免首页与任务大厅出现品牌断裂。

  - 记为后续优化，不进入首版必做范围。
- `#28` 筛选条件持久化
  - 记为后续优化，不作为首版默认行为，避免和 URL 恢复策略叠加造成状态来源混乱。

  - 采纳“任务应可被分享”的方向，但首版仅保证筛选页 URL 可分享，不新增 `id` 详情页路由。

## 五、视觉与样式复用策略

### 5.1 视觉原则

- 延续首页白底、浅灰分区、蓝色主按钮、轻阴影卡片、圆角组件
- 借鉴腾讯招聘的“左侧筛选 + 右侧列表”结构，不照搬其品牌视觉
- 提高列表页信息密度，但不改变现有品牌气质

### 5.2 CSS 拆分策略

首版采用“通用样式 + 页面专用样式”拆分，不继续把全部样式堆在一个文件里。

```text
assets/
  css/
    base.css         # 变量、重置、容器、通用排版、工具类
    components.css   # 导航、按钮、表单、标签、卡片、分页、抽屉
    home.css         # 首页专用样式
    task-hall.css    # 任务大厅专用样式
```

### 5.3 复用规则

- `base.css` 与 `components.css` 同时被 `index.html` 和 `task-hall.html` 引用
- 首页引用 `home.css`
- 任务大厅引用 `task-hall.css`
- 导航、按钮、标签、基础卡片不重复写两份

### 5.4 导航同步策略

- 首版不引入构建工具
- 使用 `assets/js/shared-layout.js` 统一渲染导航与页脚，减少 `index.html` 和 `task-hall.html` 的重复维护
- 若首版不抽取共享渲染，也必须保证导航 HTML 结构和 class 命名完全一致

## 六、任务数据模型与数据源

### 6.1 统一任务模型

```js
{
  id: "task-001",
  title: "企业官网开发",
  clientName: "某科技公司",
  summary: "需要搭建响应式官网，包含产品展示、案例、表单线索收集。",
  category: "web",
  stacks: ["react", "typescript", "nodejs"],
  deliveryMode: "milestone",
  budgetMin: 5000,
  budgetMax: 10000,
  durationDays: 14,
  deadline: "2026-04-30",
  status: "recruiting",
  updatedAt: "2026-04-03T10:00:00+08:00"
}
```

### 6.2 枚举值管理位置

首版统一放在 `assets/js/task-config.js` 中维护：

- `CATEGORY_OPTIONS`
- `STACK_OPTIONS`
- `DELIVERY_MODE_OPTIONS`
- `BUDGET_RANGE_OPTIONS`
- `DURATION_RANGE_OPTIONS`
- `STATUS_OPTIONS`
- `SORT_OPTIONS`

文件内需注释说明：“未来可由 API 返回可选项并替换本地常量”。

### 6.3 第一阶段数据源

- 使用 `assets/data/tasks.mock.json` 作为首版数据源
- 结构严格与统一任务模型一致
- 页面层通过 `task-service.js` 读取 mock 数据
- 后续接真实 API 时，仅替换 `task-service.js`，不改页面筛选和渲染逻辑

### 6.4 状态更新策略

- 首版不做实时状态推送
- 若用户当前筛选的是 `招募中`，而数据源中某任务后续已改为 `进行中`，页面只有在刷新或重新拉取后才变化

## 七、URL 参数与状态同步协议

### 7.1 参数设计

使用单值键 + 逗号分隔多选值，不使用 `[]` 数组格式。

```text
task-hall.html
  ?keyword=react
  &category=web,ai
  &stack=react,vue,typescript
  &deliveryMode=milestone,hourly
  &budgetRange=5k-10k,20k-plus
  &durationRange=8-15,16-30
  &status=recruiting,in-progress
  &sort=updated-desc
  &page=1
```

### 7.2 编码规则

- 多选参数按英文逗号分隔
- 值统一使用英文枚举值，不使用中文写入 URL
- 所有参数通过 `URLSearchParams` 编码与解析
- 搜索关键词 `keyword` 使用标准 URL 编码

### 7.3 参数校验规则

- 不在白名单内的枚举值直接忽略
- 重复值去重
- 空字符串参数视为未设置
- `page` 非数字或小于 `1` 时回退为 `1`
- 非法 `sort` 回退为默认 `updated-desc`

### 7.4 URL 更新策略

- 复选框、排序、分页变更后使用 `history.replaceState` 更新当前 URL
- 仅在显式分页跳转时允许写入新的 `page`
- 不对每次筛选都 `pushState`，避免历史记录膨胀
- 浏览器前进/后退时通过 `popstate` 重新解析 URL 并恢复页面状态

### 7.5 搜索更新策略

- 输入过程做 `300ms` 防抖，仅更新列表内存状态
- 用户按 `Enter`、输入框失焦、点击搜索按钮时，才同步写入 URL

### 7.6 URL 长度策略

- 第一阶段保留全部筛选在 URL 中，保证分享与刷新恢复
- 若未来筛选项扩展导致 URL 接近浏览器限制，再引入“URL 保留关键参数 + sessionStorage 承载扩展状态”的降级方案

## 八、筛选系统设计

### 8.1 筛选分组

- 任务类型：`web` / `mobile` / `ai` / `design-system` / `other`
- 技术栈：`react` / `vue` / `typescript` / `python` / `nodejs` / `fastapi` / `other`
- 交付模式：`milestone` / `hourly` / `hybrid`
- 预算范围：`lt-5k` / `5k-10k` / `10k-20k` / `20k-plus`
- 工期范围：`lt-8` / `8-15` / `16-30` / `30-plus`
- 任务状态：`recruiting` / `in-progress` / `closed`

### 8.2 筛选组合规则

- 组内多选使用 `OR`
  - 例如 `stack=react,vue` 表示任务命中任一技术栈即可
- 组间组合使用 `AND`
  - 例如 `category=web` 且 `status=recruiting` 表示同时满足两个分组条件
- 搜索关键词与筛选条件同样使用 `AND`

### 8.3 冲突场景处理

- 同组选多个状态不是冲突，按 `OR` 显示
  - 例如 `招募中 + 已结束` 允许并集结果
- 同组选多个预算区间不是冲突，按 `OR` 显示
  - 例如 `<5k + 20k+` 合法
- 若最终组间组合后无匹配结果，显示空状态，不自动替用户取消条件

### 8.4 区间边界定义

- 预算区间使用闭开区间，最后一档为闭区间
  - `<5k` => `[0, 5000)`
  - `5k-10k` => `[5000, 10000)`
  - `10k-20k` => `[10000, 20000)`
  - `20k+` => `[20000, +∞)`
- 工期区间同理
  - `lt-8` => `[0, 8)`
  - `8-15` => `[8, 16)`
  - `16-30` => `[16, 31)`
  - `30-plus` => `[31, +∞)`

### 8.5 已选条件展示

- 右侧标题区显示已选筛选标签
- 每个标签支持单独移除
- 提供 `清空筛选` 按钮

## 九、筛选组件 HTML 结构约定

```html
<aside class="filter-sidebar" aria-label="任务筛选">
  <div class="filter-sidebar__header">
    <h2 class="filter-sidebar__title">筛选任务</h2>
    <button type="button" class="filter-sidebar__reset">清空筛选</button>
  </div>

  <section class="filter-group" data-filter-type="category">
    <button
      type="button"
      class="filter-group__toggle"
      aria-expanded="true"
      aria-controls="filter-group-category"
    >
      任务类型
    </button>
    <div id="filter-group-category" class="filter-group__content">
      <label class="filter-option">
        <input type="checkbox" name="category" value="web" />
        <span>Web 开发</span>
      </label>
    </div>
  </section>

  <section class="filter-group" data-filter-type="stack">
    <button
      type="button"
      class="filter-group__toggle"
      aria-expanded="true"
      aria-controls="filter-group-stack"
    >
      技术栈
    </button>
    <div id="filter-group-stack" class="filter-group__content"></div>
  </section>
</aside>
```

### 9.1 结构规则

- 筛选组统一使用 `.filter-group`
- 组标题按钮统一使用 `.filter-group__toggle`
- 内容容器统一使用 `.filter-group__content`
- 单项统一使用 `.filter-option`
- 复选框的 `name` 必须与 URL 参数键一致

## 十、任务列表与分页设计

### 10.1 顶部操作区

- 搜索框
- 当前结果数
- 排序下拉
- 已选条件标签
- 清空筛选

### 10.2 任务卡片字段

- 标题
- 委托方名称
- 任务简介
- 标签：类型、技术栈、交付模式
- 预算
- 工期
- 截止时间
- 状态
- 最近更新时间
- 操作按钮：`查看任务`

### 10.3 分页规则

- 默认每页 `12` 条
- 只渲染当前页卡片，不渲染全部结果后再隐藏
- 当筛选、搜索、排序任一条件变化时，`page` 强制重置为 `1`

### 10.4 性能规则

- 首版使用分页控制 DOM 数量
- 不引入虚拟滚动
- 目标是 1000 条 mock 任务数据下，筛选响应时间 `< 500ms`

## 十一、加载态、错误态、空状态设计

### 11.1 加载态

- 搜索或筛选触发数据请求时，右侧列表区显示 `6` 个骨架卡片
- 顶部结果数区域显示 `加载中...`

### 11.2 错误态

- 使用错误提示卡片替代列表
- 文案：
  - 标题：`任务加载失败`
  - 说明：`暂时无法获取任务列表，请稍后重试。`
  - 按钮：`重新加载`

### 11.3 空状态

- 显示简洁插图或空文件夹图标
- 文案：
  - 标题：`没有找到符合条件的任务`
  - 说明：`试试减少筛选条件，或更换关键词。`
  - 按钮：`清空筛选`

## 十二、移动端抽屉交互

### 12.1 打开方式

- 页面顶部显示 `筛选任务` 按钮
- 点击后从右侧或底部弹出全屏抽屉

### 12.2 抽屉行为

- 打开时锁定背景滚动
- 点击遮罩可关闭
- 顶部提供关闭按钮
- 底部固定两个按钮：
  - `取消`
  - `应用筛选`

### 12.3 生效规则

- 移动端抽屉内的勾选先写入临时状态
- 用户点击 `应用筛选` 后再刷新列表并同步 URL
- 用户点击 `取消` 或遮罩关闭时，不提交临时修改

### 12.4 桌面端与移动端差异

- 桌面端筛选实时生效
- 移动端筛选批量生效

## 十三、安全与可维护性约束

### 13.1 XSS 与输入安全

- 搜索关键词仅作为纯文本处理
- 不使用 `innerHTML` 直接渲染用户输入
- 所有动态文本通过 `textContent` 注入

### 13.2 URL 参数安全

- 仅解析白名单参数
- 非法枚举值忽略
- 超出范围的分页值回退默认值

### 13.3 维护性约束

- 枚举值集中管理
- 数据访问单独封装
- 共享布局与共享组件样式抽离
- 页面脚本不得写死筛选项 DOM 文本和枚举值两份来源

### 13.4 文档同步要求

实现完成后需同步检查下列文档是否需要更新：

- `设计文档/任务大厅页面设计方案.md`
- `设计文档/CLAUDE.md`
- `README.md`

若实现过程中调整了全站共享样式结构，也应补充更新首页设计文档中的样式引用说明。
