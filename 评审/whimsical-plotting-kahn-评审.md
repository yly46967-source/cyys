# `whimsical-plotting-kahn.md` 技术评审

## 评审结论

当前方案**不建议直接进入实施**。主要问题不是 UI 细节，而是基础契约没有收口：现有服务层/API/数据模型与方案定义明显不一致，私信与任务状态模型不闭合，权限与回滚方案存在错误假设，后续非常容易在接入真实数据时发生返工。

---

## 一、理解错误的需求

### 1. 私信需求被错误建模成“通知列表”

- 方案位置：`3.3.2`、`5.3.1`、`5.3.2`
- 问题：方案把 `private` 私信和 `system/task` 通知放进同一个扁平消息模型，只定义了 `title/content/fromUserId/fromUserName`，没有 `conversationId`、会话对端、最后一条消息、会话级未读数、消息方向、已删除会话等字段。
- 影响：如果“用户私信”是真正的站内私聊，这个模型无法支撑后续会话列表、详情页、回复、已读回执，后面一定返工。
- 判断：这是需求理解错误，不是实现细节遗漏。

### 2. 开发者用户中心与任务大厅职责混淆

- 方案位置：`2.2` 中“开发者版主要功能 = 浏览任务、参与竞标、管理我的参与”
- 现有实现依据：仓库已存在 `task-hall.html`、`assets/js/task-hall.js`、`assets/js/task-service.js`
- 问题：用户中心核心应是“我的参与/我的状态”，而“浏览任务、参与竞标”已经是任务大厅职责。当前写法会让用户中心与任务大厅重复承载“任务浏览/竞标入口”。
- 影响：导航、入口、筛选、排序、测试矩阵会重复，后续极易出现两个页面逻辑不一致。

### 3. 角色访问与实名认证流程的前提被说得过于乐观

- 方案位置：`2.3`、`7.1`、`8.1.1`
- 现有实现依据：
  - `assets/js/auth-guard.js:19-27` 仅保护 `profile-client.html` / `profile-developer.html`
  - `assets/js/auth-guard.js:96-103` 角色不匹配时跳 `index.html`
  - `assets/js/auth-guard.js:105-113` 实名认证失败跳 `real-name-auth.html`，但该页在代码中标注“暂不实现”
- 问题：方案写成“确认路由守卫配置正确”过轻，实际这里不是配置确认，而是行为定义不一致。
- 影响：如果按方案测试“自动跳正确角色页”“实名认证拦截可用”，现在的基础设施并不满足，实施期会反复改守卫逻辑。

## 二、遗漏的边界条件

### 4. 私信/通知的生命周期边界未定义

- 方案位置：`5.3`
- 缺失点：
  - 系统消息是否允许删除
  - 被删除的任务消息点击后跳哪里
  - 私信是否按会话聚合
  - 标记已读是“打开详情即已读”还是“进入会话即已读”
  - 删除后未读计数是否立即回补
- 影响：消息中心最容易因为这些细节返工。

### 5. 任务操作的并发与状态竞争未定义

- 方案位置：`5.2.1`、`5.2.2`、`7.3`
- 缺失点：
  - 列表加载后，任务已被别人关闭/雇佣/拒绝时如何处理
  - 重复点击“关闭任务/撤销竞标/标记完成”是否幂等
  - 乐观更新失败后 UI 如何回滚
  - 客户确认完成与开发者标记完成的状态流转谁为准
- 影响：这些不是边角 case，而是任务管理主流程。

### 6. 资料编辑的脏数据与取消路径未定义

- 方案位置：`5.1.2`、`7.3`
- 缺失点：
  - 关闭模态框、刷新页面、切换 Tab、返回上一页时是否拦截未保存修改
  - 头像压缩失败、上传取消、格式伪装、尺寸过小/过大如何处理
  - 客户 `company`、开发者 `skills/profile` 缺字段时如何回填
- 影响：资料编辑看似简单，实际最容易出现“用户以为保存了/实际没保存”的投诉。

### 7. 登录返回路径只写了“目标 URL”，没定义 query/hash 恢复

- 方案位置：`7.1`
- 现有实现依据：`assets/js/auth-guard.js:91-92,134-155` 仅保存基础路径
- 问题：用户如果从用户中心的某个 Tab、筛选状态、锚点进入登录页，登录后是否恢复原状态没有定义。
- 影响：如果后续任务列表和消息中心采用 hash/query 表达内部状态，会出现登录后丢上下文。

### 8. 跨标签页同步只提“同步”，没有定义冲突策略

- 方案位置：`5.3.3`、`6.1`、`7.3`
- 现有实现依据：`assets/js/auth-state.js` 已有跨标签页广播机制
- 缺失点：
  - 两个标签页同时删除同一条消息怎么办
  - 一个标签页已读，另一个标签页还在详情页时如何刷新
  - 浏览器后台标签页是否暂停轮询/心跳
- 影响：未读计数很容易出现“闪回”“负数”“回跳”。

## 三、可能导致返工的设计问题

### 9. API 契约与现有服务层不一致

- 方案位置：`2.3`、`3.4.2`
- 现有实现依据：
  - `assets/js/auth-config.js:52-65`
  - `assets/js/auth-service.js:89-229`
- 冲突点：
  - 方案写 `GET /api/users/:id`，现有是 `GET /api/users/profile`
  - 方案写 `GET /api/users/:id/tasks`，现有是 `GET /api/tasks/my`
  - 方案写 `GET /api/users/:id/messages`，现有是 `GET /api/messages`
  - 方案未写 `PUT /api/messages/read-all`，现有服务已经有
- 影响：一边说“复用现有 auth-service”，一边又定义另一套接口，实施时要么推翻现有服务，要么再做一层适配，必然返工。

### 10. 用户数据模型与现有 Mock/认证模型冲突

- 方案位置：`3.3.1`
- 现有实现依据：`assets/data/auth-mock.js:16-53`
- 冲突点：
  - 方案把 `company` 定义成字符串，现有是 `company.name/creditCode`
  - 方案把开发者资料定义为顶层 `bio/website/skills`，现有是 `profile.bio/profile.portfolioUrl/profile.skills`
  - 方案统计字段为 `tasksPosted/tasksParticipated/...`，现有是 `postedTasks/totalBudget/inProgressTasks/completedTasks/totalEarnings/rating`
- 影响：资料表单、概览卡片、保存接口、Mock 数据都要做字段映射；如果不先定统一模型，越做越乱。

### 11. 任务状态模型与现有任务系统不闭合

- 方案位置：`5.2.1`、`5.2.2`、`5.4`
- 现有实现依据：`assets/js/task-config.js:80-84`
- 冲突点：
  - 现有任务状态只有 `recruiting / in-progress / closed`
  - 方案新增了客户侧 `已完成`，开发者侧 `已接受 / 已完成 / 已拒绝 / 竞标中`
  - 方案没有说明“任务状态”和“我的申请状态”如何组合映射
- 影响：筛选 Tab、统计卡片、空状态、接口返回、测试用例都会被这组状态牵连。

### 12. 状态管理被拆成两套，存在单一事实源冲突风险

- 方案位置：`3.2`、`5.3.3`
- 现有实现依据：`assets/js/auth-state.js` 已负责用户态、事件订阅、跨标签同步
- 问题：方案再引入 `profile-state.js`，但没有说明它与 `auth-state.js` 的边界，尤其是当前用户资料、未读数、登出清理、跨标签同步谁说了算。
- 影响：同一份用户数据如果在两个 store 里维护，迟早会出现 navbar、侧边栏、模态框显示不一致。

### 13. 双页面方案没有给出共享抽象，维护成本会翻倍

- 方案位置：`4.1`、`10.2`
- 问题：`profile-client.html` 和 `profile-developer.html` 结构高度相似，但方案只拆了两个页面入口，没有说明共享模板、配置化菜单、差异化动作如何抽象。
- 影响：DOM 结构、样式、a11y、响应式、测试都要维护两份。中后期改一个漏一个的概率很高。

### 14. 回滚与数据回滚方案存在错误假设

- 方案位置：`9.1`、`9.2`
- 问题：
  - `git checkout v1.0.0` 和 `git push origin master --force` 不是可接受的常规发布回滚策略
  - “清除浏览器缓存后自动恢复 LocalStorage 数据”是错误表述，清缓存本身就会丢数据
  - 没有任何前端本地 schema 兼容/迁移说明
- 影响：一旦资料结构升级或未读计数格式变化，线上回退会留下脏数据。

## 四、测试不足

### 15. 缺少接口契约与数据映射测试

- 方案位置：`8.1`、`8.2`
- 缺口：没有验证“现有 auth-service/auth-mock/task-config”与新页面模型的映射是否正确。
- 后果：最先爆雷的一定是字段名不一致、状态值不一致、空字段崩溃。

### 16. 缺少认证失败与 Token 刷新链路测试

- 方案位置：`8.1.1`、`8.2`
- 现有实现依据：`assets/js/http-client.js` 有 401 自动刷新、刷新失败登出队列逻辑
- 缺口：
  - 401 后自动刷新成功
  - 刷新失败后回登录页
  - 多请求并发等待刷新
  - 登录后回跳用户中心原页面
- 后果：用户中心是登录后高频页面，不测这条链路风险很高。

### 17. 缺少状态竞争与幂等性测试

- 方案位置：`8.1.3`、`8.1.4`、`8.3`
- 缺口：
  - 关闭中的任务再次关闭
  - 已撤销竞标再次撤销
  - 两标签页同时删除同一消息
  - 任务状态变更后当前列表自动刷新
- 后果：真实环境下会出现大量“第二次操作”的脏状态。

### 18. 缺少模态框/抽屉的可访问性与交互测试

- 方案位置：`8.6`
- 缺口：
  - 焦点陷阱
  - `Esc` 关闭
  - 抽屉打开后背景滚动锁定
  - 键盘操作头像上传/Tab 切换/消息列表
- 后果：方案写了可访问性，但没有覆盖真正有风险的交互组件。

### 19. 缺少安全测试

- 方案位置：`6.3`、`8.x`
- 缺口：
  - 恶意 `javascript:` / `data:` URL 注入
  - 头像文件 MIME 伪装
  - 超长简介/异常 Unicode/脚本片段
  - 外链打开的 `noopener/noreferrer`
- 后果：当前安全措施写得过于抽象，测试也没有落到攻击面。

### 20. 性能指标缺少测量方法和约束前提

- 方案位置：`8.4`
- 问题：只给了 `<500ms`、`<2s` 这类数字，没有说明设备基线、数据规模、缓存条件、是否冷启动、是否含图片与 Mock 拦截开销。
- 后果：这些指标无法稳定验收，也无法作为是否达标的依据。

## 五、性能、安全、维护性重点问题

### 21. `textContent` 不是完整的 XSS 防线

- 方案位置：`6.3`
- 问题：简介字段用 `textContent` 还不够，`website`、`avatar`、跳转链接、图片 `src`、上传文件名都不是 `textContent` 场景。
- 必须补充：
  - URL scheme allowlist
  - 外链属性约束
  - 文件类型/MIME/尺寸校验
  - 服务端二次校验

### 22. “轮询 + storage + 心跳”过度复杂，且可能带来性能负担

- 方案位置：`6.1`、`6.2`
- 问题：当前页面规模并不大，方案却同时引入轮询、跨标签同步、心跳检测，没有说明触发条件、暂停策略、退避策略。
- 影响：前台/后台多标签页都会放大资源消耗，且实现复杂度明显高于当前需求。

### 23. 虚拟滚动/分页阈值没有和真实数据来源绑定

- 方案位置：`6.1`、`7.4`、`11.2`
- 问题：方案提出“100 条消息上虚拟滚动”，但当前接口契约里并没有 cursor/分页约束，也没有说明是否真会有这么长的列表。
- 影响：容易先做重组件，后面又因服务端契约变化重做。

### 24. 维护面正在快速扩大，但方案没有约束“共享优先”

- 方案位置：整体
- 问题：页面、组件、Mock、服务、状态、样式全部新增，但没有说明哪些必须共享、哪些允许角色差异化。
- 影响：后续所有变更都可能横跨 2 个页面 + 3 套数据源 + 2 套状态。

## 六、分级结论

### 必须修改项

1. 统一 API 契约，明确是复用现有 `auth-service/auth-config`，还是废弃后重建，不能两套并存。
2. 统一用户资料与统计数据模型，先解决 `auth-mock`/`auth-service`/方案字段不一致问题。
3. 补完整任务状态机，明确“任务状态”和“我的参与状态”的映射关系，再设计 Tab 与统计卡片。
4. 明确“私信”是否为真实会话；如果是，必须重做消息模型，不能沿用扁平通知结构。
5. 重新定义权限/角色/实名认证流程，并与 `auth-guard.js` 的实际行为对齐。
6. 明确单一事实源，避免 `auth-state` 与 `profile-state` 重复持有用户核心状态。
7. 修正安全方案，至少覆盖 URL 注入、头像上传校验、外链安全、服务端权限兜底。
8. 修正回滚与本地数据回退方案，去掉错误的 Git/LocalStorage 假设。
9. 增补契约测试、认证刷新测试、并发幂等测试、安全测试，否则验收标准不成立。

### 建议修改项

1. 收紧用户中心范围，避免与 `task-hall.html` 重复承担“浏览任务/参与竞标”职责。
2. 如果继续保留双页面，必须显式规定共享模板/配置层，避免两页分叉。
3. 补充消息、任务、资料编辑的冲突处理策略和错误恢复策略。
4. 补充模态框、抽屉、移动端导航的焦点管理与键盘行为说明。
5. 对性能指标补充测量环境、样本规模和验收方法。

### 可选优化项

1. 未读消息优先做“聚合计数接口/变更事件”，再决定是否需要前端心跳。
2. 虚拟滚动可以延后到真实数据规模确认之后，再按热点场景引入。
3. 如果未来用户中心功能继续扩展，可以考虑单页 + 角色配置化，而不是持续复制页面。

---

## 总体判断

这份方案的主要问题不是“写得不够多”，而是**现有基础设施与新方案之间的契约没有对齐**。如果不先把 API、数据模型、状态归属、权限流转这四件事定死，后面无论先做页面还是先做 Mock，都会在接真实数据时返工。
