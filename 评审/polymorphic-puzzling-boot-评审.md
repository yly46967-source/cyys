# `polymorphic-puzzling-boot-v3-final.md` 技术评审

## 评审结论

v3.0 比 v2.0 多了一层“逐条回应评审”的结构，表面上更完整，但**仍不适合标记为“最终版，可进入实施阶段”**。  
当前最大的问题不是缺少页面草图，而是方案里仍有多处“自称已解决、实际未闭合”的技术前提，继续按这份文档推进，返工点会集中出现在：

1. Mock 路由机制仍无法支撑动态路径。
2. 多处代码示例调用当前代码库不存在的 API。
3. 资料更新链路和状态同步链路的迁移方案仍未闭合。
4. 消息详情页的状态持久化、深链鉴权、富文本安全仍未收口。

---

## 一、理解错误的需求

### 1. `[必须修改]` 把“已采纳评审意见”当成“已具备可实施方案”，结论过早

- 方案位置：
  - 文档标题“最终修订版 v3.0”
  - 文末“文档状态：最终版，可进入实施阶段”
- 问题：
  - 方案仍保留多处当前代码基线不支持的接口假设。
  - 多个关键实现点只停留在口头“完全采纳”，没有真正闭合到现有代码结构。
- 影响：
  - 这会误导执行方低估工作量，提前进入开发，最后在联调阶段集中返工。

### 2. `[必须修改]` “暂停密码功能”与文档中实际给出的实现假设仍不一致

- 方案位置：
  - 第一部分“问题 1 / 2 已完全采纳”
  - 边界条件 3.2 的实现示例
- 问题：
  - 文档一边说密码相关功能暂停，一边又给出 `handlePasswordChange()` 的具体实现示例，并直接使用 `authState.clearAuth()`
  - 但当前代码里并不存在 `authState.clearAuth()`
- 影响：
  - 这会让实施人误判为“密码能力只是暂缓上线，但基础实现路径已明确”
  - 实际上连状态管理 API 都未收口

### 3. `[必须修改]` 账户安全页被定义为“最小只读版”，但最小数据模型仍未真正定义到现有基线

- 方案位置：
  - 2.2.3 账户安全设置页
  - 3.5 空态和失败态
- 问题：
  - 方案写了 `GET /api/users/security/status`
  - 示例直接用 `authState.getCurrentUser()` 生成数据
  - 当前代码中没有 `getCurrentUser()` 方法，`auth-mock.js` 也不是一个可直接访问 `authState` 的模块上下文
- 影响：
  - 这说明“最小只读版”的底层数据来源并没有真正建模，还是停留在伪代码层面

## 二、遗漏的边界条件

### 4. `[必须修改]` 新增页面的鉴权范围没有纳入实施清单

- 方案位置：
  - 文件改动清单 4.1 / 4.2 / 4.3
- 当前现状：
  - `assets/js/auth-guard.js` 的 `PROTECTED_ROUTES` 里没有 `profile-edit.html`、`message-detail.html`、`security-settings.html`
- 问题：
  - 方案多次写“未登录时 auth-guard 自动重定向并保存返回路径”
  - 但文件改动清单里根本没有 `auth-guard.js`
- 影响：
  - 新页面深链进入时不会被保护，登录回跳链路也不会真正成立

### 5. `[建议修改]` 资料编辑页和消息详情页的返回路径仍然是写死的，缺少来源上下文

- 方案位置：
  - 资料页示例中出现 `window.location.href = 'profile-client.html'`
  - 消息详情深链 fallback 里写死 `profile-client.html#messages`
- 问题：
  - 没有保留角色信息、原入口、原 section、原筛选条件
  - 对开发者角色和从其他页面进入的场景都不成立
- 影响：
  - 后续只要用户提出“返回上一状态”的需求，就要重改导航模型

### 6. `[建议修改]` “未保存离开拦截”没有覆盖站内脚本跳转和统一导航入口

- 方案位置：
  - 5.3 独立资料编辑页的离开拦截
- 问题：
  - 方案只覆盖了 `beforeunload` 和 `popstate`
  - 对站内按钮跳转、侧边导航切换、脚本 `window.location.href` 跳转没有统一策略
- 影响：
  - 真正落地到站内多入口后，拦截行为会不一致

## 三、可能导致返工的设计问题

### 7. `[必须修改]` Mock 路由改成 `{METHOD} /path` 仍然无法支持 `GET /api/messages/{id}`

- 方案位置：
  - 0.2 修复 Mock API 路由分发
  - 2.2.2 / 4.2.2 中新增 `GET /api/messages/{id}`
- 问题：
  - 当前方案只把分发逻辑从 `pathname` 改成 `method + pathname`
  - 这仍然是**精确匹配**
  - 实际请求路径是 `/api/messages/msg-001`，不会等于 `/api/messages/{id}`
- 影响：
  - 消息详情页、标记已读、删除消息这些带参数路径在 Mock 层仍然不可用
- 结论：
  - 必须补动态路径匹配规则，而不是只换 key 格式

### 8. `[必须修改]` `getAdjacentMessages()` 的示例仍然与当前 `message-service` 契约不匹配

- 方案位置：
  - 第一部分“问题14已完全采纳”
  - 2.2.2 / 示例代码
- 当前现状：
  - `messageService.getMessages()` 返回的是 `{ messages, unreadCount, pagination }`
- 方案示例：
  - `const messages = await this.getMessages();`
  - `const sortedMessages = messages.sort(...)`
- 问题：
  - 这里排序的是对象，不是数组
- 影响：
  - 说明消息详情页服务层契约仍然没有真正设计完，按文档开发后必然回头改接口

### 9. `[必须修改]` 账户安全页的 Mock 示例在当前 `auth-mock.js` 里无法成立

- 方案位置：
  - 2.2.3 安全状态 Mock API 示例
- 示例内容：
  - `'GET /api/users/security/status': () => { const user = authState.getCurrentUser(); ... }`
- 当前现状：
  - `assets/data/auth-mock.js` 当前不是依赖 `auth-state.js` 的模块
  - 文件内没有 `authState` 引用
  - `auth-state.js` 也没有 `getCurrentUser()`
- 影响：
  - 这不是“后面再细化”的问题，而是示例本身在当前代码基线里就不可执行

### 10. `[必须修改]` 删除 `auth-service.updateProfile()` 的迁移路径仍未闭合

- 方案位置：
  - 第一部分“问题11已完全采纳”
  - 文件清单 4.1 写“移除 updateProfile 方法”
- 当前现状：
  - `assets/js/auth-state.js` 的 `onUpdateUser()` 仍直接依赖 `authService.updateProfile()`
- 问题：
  - 文件清单里没有同步修改 `assets/js/auth-state.js`
  - 也没有定义是保留兼容层，还是改成 `userService` 唯一入口
- 影响：
  - 如果按清单直接删方法，认证状态管理会先断

### 11. `[必须修改]` 消息状态持久化方案仍然缺失，详情页设计仍然会和列表页状态打架

- 方案位置：
  - 2.2.2 消息详情页
  - 6.2.2 / 6.2.3 测试清单
- 当前现状：
  - `assets/js/message-service.js` 的 `_mockMarkMessageRead()` 只返回成功，不修改实际存储
  - `_mockDeleteMessage()` 也不持久化删除状态
- 问题：
  - 文档加入了“自动标记已读”“删除后返回列表”“其他标签页同步”
  - 但没有定义这些状态的唯一数据源和持久化位置
- 影响：
  - 刷新、回退、跨页切换后状态会回滚
  - 测试能列出来，不等于设计已经成立

### 12. `[建议修改]` `pushState + popstate` 方案仍然是高维护成本的 hack

- 方案位置：
  - 第一部分“问题7 已完全采纳”
  - 5.3 的实现示例
- 问题：
  - 进入页面就 `pushState`
  - 用户取消后退时再 `pushState`
  - 这会污染历史栈
- 影响：
  - 后续如果接入更多独立页、抽屉页、导航同步，这套逻辑会越来越难维护

### 13. `[建议修改]` 文档编号和排期仍不一致

- 方案位置：
  - 6.1 开发顺序建议
  - `Day 5: 基础工作 0.5-0.7`
- 问题：
  - 正文只有 `0.1-0.4`
  - 边界条件定义被写在第 3 章
  - `0.7` 在文档中并不存在
- 影响：
  - 任务拆分、工时预估和验收口径会继续混乱

## 四、测试不足

### 14. `[必须修改]` 缺少动态路径匹配测试

- 需要覆盖：
  - `GET /api/messages/:id`
  - `PUT /api/messages/:id/read`
  - `DELETE /api/messages/:id`
  - 同一路径不同 method 的 handler 命中
- 原因：
  - 这是 v3 仍然最核心的实现风险，但测试清单没有把“动态匹配机制”单独测穿

### 15. `[必须修改]` 缺少新增页面的鉴权 / 深链 / 登录回跳测试闭环

- 需要覆盖：
  - 未登录直达 `profile-edit.html`
  - 未登录直达 `message-detail.html?id=...`
  - 登录后返回原始 query / hash
- 原因：
  - 方案声称支持深链，但没有把 `auth-guard` 变更纳入清单，也没有把这一整条链路测透

### 16. `[必须修改]` 缺少资料更新链路迁移测试

- 需要覆盖：
  - `auth-service.updateProfile` 删除或保留兼容层后，`auth-state` 是否仍正常
  - 用户资料更新后 `auth-state.currentUser` 是否同步
  - 跨标签页资料更新广播是否正常
- 原因：
  - v3 继续推动统一资料契约，但对状态同步链路没有测试闭环

### 17. `[建议修改]` 缺少富文本安全测试

- 需要覆盖：
  - 消息内容包含 HTML
  - 消息内容包含脚本片段
  - 非法标签和危险属性是否被剔除
- 原因：
  - 方案明确写了“支持富文本消息内容”，但没有任何对应安全测试

## 五、性能 / 安全 / 维护性问题

### 18. `[必须修改]` 富文本消息内容仍然没有 XSS 防护方案

- 方案位置：
  - 2.2.2 写“支持富文本消息内容”
- 问题：
  - 方案没有定义白名单、转义策略或 sanitizer
- 影响：
  - 一旦消息内容来自后端或富文本模板，详情页会引入明显的 XSS 风险
- 结论：
  - 本版要么降级为纯文本渲染，要么必须补充消毒方案

### 19. `[建议修改]` 单一事实源仍未真正建立

- 当前至少还有三套数据来源：
  - `auth-state.currentUser`
  - `user-service` 的本地 Mock 存储
  - `auth-mock.js` 的 `MOCK_USERS`
- 问题：
  - v3 继续写“统一契约已采纳”，但没有定义谁是真正主源、谁负责回写、谁只是缓存
- 影响：
  - 后续每加一个资料字段，都仍然可能多处同步、多处返工

### 20. `[建议修改]` 消息轮询问题仍未纳入本轮设计

- 当前现状：
  - `assets/js/message-center.js` 初始化后立即启动 30 秒轮询
- 问题：
  - v3 新增详情页，但没有重新定义轮询生命周期
- 影响：
  - 页面越多，未读数同步和无效轮询的复杂度越高

## 六、分级结论

### 必须修改项

1. 取消“最终版，可进入实施阶段”的结论，先把动态路由、状态同步、鉴权和安全问题收口。
2. 把 Mock 路由从 `{METHOD} /path` 精确匹配升级为支持动态路径参数的匹配机制。
3. 校正文档中所有调用不存在 API 的实现示例，至少包括 `authState.clearAuth()`、`authState.getCurrentUser()` 这几处。
4. 把 `auth-guard.js` 和 `auth-state.js` 明确纳入文件改动清单；否则新页面鉴权和资料迁移都不成立。
5. 明确资料更新链路的唯一入口和迁移顺序，不能只写“移除 `auth-service.updateProfile()`”。
6. 为消息详情页补齐状态持久化方案，不能只新增 `getMessageById()`。
7. 为“富文本消息内容”补齐安全方案；如果本轮不做，就在需求里明确降级为纯文本。

### 建议修改项

1. 不要把返回路径写死到 `profile-client.html`，至少保留角色和来源上下文。
2. 调整离开拦截方案，减少 `pushState` 历史栈污染。
3. 修正文档编号和开发排期，去掉不存在的 `0.7`。
4. 明确消息轮询只在消息区域可见时启动。

### 可选优化项

1. 为资料页和消息详情页加一层 view-model，减少 UI 对底层存储结构的直接耦合。
2. 给消息详情页增加来源参数，例如 `from=messages&filter=unread`，方便后续保留返回状态。
3. 把新增页面统一纳入 `*-main.js + auth-guard + shared-layout` 的初始化模式。

---

## 最终判断

这份 v3 文档比前一版更像“回应评审的总结稿”，但还不是“可直接实施的设计稿”。  
当前最关键的风险不是 UI，而是以下四个基础面仍未闭合：

1. Mock 动态路由
2. 资料更新与认证状态同步
3. 消息状态持久化与新页面鉴权
4. 富文本与安全状态模型

在这四点修正之前，不建议按“最终版”推进实施。
