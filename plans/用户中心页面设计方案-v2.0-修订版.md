# 用户中心页面设计方案 v2.0（修订版）

## 文档说明

本文档是针对 v1.0 方案的修订版本，基于技术评审意见进行了全面修正。

**主要修订内容**：
1. 统一 API 契约，与现有 auth-service/auth-config 完全对齐
2. 重新设计数据模型，适配现有 auth-mock.js 结构
3. 对齐任务状态模型，仅扩展 applicationStatus 字段
4. 移除重复的状态管理，复用现有 auth-state.js
5. 补充完整的边界条件、并发处理、安全方案
6. 修正回滚方案，移除错误假设
7. 补充完整测试方案

---

## 一、文档目标

本文档用于定义 TechCraft 项目的用户中心页面（客户版和开发者版）的页面结构、功能模块、数据组织和实现约束，作为 `profile-client.html` 和 `profile-developer.html` 的设计与前端实现依据。

---

## 二、需求理解

### 2.1 核心需求

用户中心是登录后的用户专属页面，提供以下核心功能：

1. **个人资料管理**：展示和编辑用户基本信息、头像、简介等
2. **任务管理**：
   - 客户：查看和管理自己发布的任务
   - 开发者：查看和管理自己参与/竞标的任务（**仅管理"我的参与状态"，不包括浏览新任务**）
3. **消息中心**：**系统通知**（本阶段不实现私信会话功能）
4. **权限控制**：必须登录后才能访问，根据角色显示不同页面

### 2.2 用户角色区分

| 角色 | 页面文件 | 主要功能 |
|------|----------|----------|
| 客户 (client) | `profile-client.html` | 管理我发布的任务（查看、编辑、关闭、查看竞标） |
| 开发者 (developer) | `profile-developer.html` | 管理我参与的任务（竞标、撤销、查看状态） |

**重要说明**：用户中心不承担"浏览任务、参与竞标"的职责，这些功能属于 `task-hall.html` 任务大厅。

### 2.3 与现有系统集成

#### 2.3.1 认证系统集成

**复用现有模块**：
- `auth-state.js` - 用户认证状态管理（**不新增 profile-state.js**）
- `auth-guard.js` - 路由守卫
- `auth-service.js` - 认证服务
- `auth-config.js` - 认证配置

**权限流程明确定义**：
```
访问 profile-client/profile-developer
    ↓
检查登录状态 (auth-state.isLoggedIn())
    ↓ 未登录
跳转登录页 (login.html?redirect=<当前完整URL含query/hash>)
    ↓ 已登录
检查角色匹配 (auth-state.getUserRole())
    ↓ 角色不匹配
跳转到正确角色页面
    ↓ 角色匹配
检查实名认证状态 (如需要)
    ↓ 未认证/认证失败
显示提示，限制特定操作（不强制跳转，允许浏览）
```

#### 2.3.2 样式系统集成

- **复用**：`base.css`、`components.css`
- **新增**：`profile.css`（用户中心专用样式）

#### 2.3.3 任务系统集成

**复用现有模块**：
- `task-config.js` - 任务配置（状态枚举等）
- `task-service.js` - 任务服务
- `task-hall.js` - 任务大厅交互（部分可复用）

**任务状态对齐**：
- 现有状态：`recruiting` | `in-progress` | `closed`
- 新增扩展：`applicationStatus`（仅用于开发者视角）= `pending` | `accepted` | `rejected` | `withdrawn`

#### 2.3.4 布局系统集成

- **复用**：`shared-layout.js`（导航栏、页脚）
- **扩展**：导航栏增加未读消息红点

## 三、技术方案

### 3.1 技术栈

- **前端框架**：原生 JavaScript (ES6+ 模块)
- **样式**：CSS3 (CSS 变量、Flexbox、Grid)
- **数据存储**：LocalStorage/SessionStorage (认证信息) + Mock 数据 (开发阶段)
- **HTTP 请求**：Fetch API (复用现有 http-client.js)

### 3.2 架构设计

```
用户中心页面架构
├── 路由守卫层 (auth-guard.js) - 现有，可能需扩展
│   └── 检查登录状态、角色权限、实名认证状态
├── 数据服务层 (新增/扩展)
│   ├── user-service.js - 用户数据服务（适配现有 auth-service）
│   ├── message-service.js - 消息数据服务
│   └── my-tasks-service.js - 我的任务数据服务（适配现有 task-service）
├── 状态管理层 (复用现有)
│   └── auth-state.js - 用户认证状态管理（不新增 profile-state.js）
├── 组件层 (新增)
│   ├── profile-header.js - 个人资料头部组件
│   ├── task-list-manager.js - 任务列表管理组件
│   ├── message-center.js - 消息中心组件
│   └── profile-editor.js - 个人资料编辑组件 (模态框)
└── 页面入口层 (新增)
    ├── profile-client-main.js - 客户页面入口
    └── profile-developer-main.js - 开发者页面入口
```

**状态管理单一事实源**：
- **用户认证状态**：由 `auth-state.js` 管理（不重复）
- **用户资料数据**：由 `user-service.js` 获取，组件内部缓存
- **未读消息计数**：由 `auth-state.js` 扩展存储，跨标签页同步复用现有机制
- **任务列表数据**：由 `my-tasks-service.js` 获取，组件内部缓存

#### 3.3.2 消息数据模型（修订版）

**设计原则**：本阶段仅实现单向系统通知，不实现私信会话功能。

```javascript
{
  id: "msg-001",
  type: "system", // system | task（暂不实现 private）
  title: "任务状态变更通知",
  content: "您发布的任务「企业官网开发」已有新的竞标",
  taskId: "task-001",      // 关联任务ID（可选）
  taskTitle: "企业官网开发", // 关联任务标题（可选，用于跳转显示）
  actionType: "task_new_bid", // 消息动作类型，用于跳转路由
  actionUrl: "/task-hall?task=task-001", // 点击后跳转的URL
  isRead: false,
  createdAt: "2026-04-07T14:30:00+08:00",
  deletedAt: null,         // 软删除时间戳（null表示未删除）

  // === 生命周期字段（新增）===
  lifecycle: {
    canDelete: true,       // 是否允许删除
    deleteAction: "remove", // 删除后行为：remove|mark-read|archive
    jumpIfTaskDeleted: "task-hall" // 任务被删除后的跳转目标
  }
}
```

**消息类型定义**：
| type | 说明 | 可删除 | 点击行为 |
|------|------|--------|----------|
| system | 系统通知（实名认证、平台公告等） | 是 | 打开详情或跳转 |
| task | 任务相关（新竞标、被雇佣等） | 是 | 跳转任务详情或任务大厅 |

**后续阶段扩展**：私信会话功能留待后续阶段，届时需引入会话模型：

```javascript
  lastMessage: { ... },

  // ...
}
```

**任务状态与申请状态的关系**：
| 任务状态 | 我的申请状态 | 说明 |
|----------|--------------|------|
| recruiting | pending | 竞标中 |
| recruiting | accepted | 已被雇佣 |
| recruiting | rejected | 竞标被拒绝 |
| recruiting | withdrawn | 已撤销竞标 |
| in-progress | accepted | 进行中 |
| closed | accepted | 任务已关闭 |

#### 3.4.2 预留真实 API 接口（修订版，对齐现有）

```
# === 用户相关（对齐现有 auth-service）===
GET    /api/users/profile              # 获取当前用户资料
PUT    /api/users/profile              # 更新用户资料
POST   /api/users/avatar               # 上传头像

# === 任务相关（对齐现有 task-service）===
GET    /api/tasks/my                   # 获取我的任务（客户/开发者根据角色自动区分）
PUT    /api/tasks/:id/close            # 关闭任务
POST   /api/tasks/:id/apply            # 竞标任务
DELETE /api/tasks/:id/application      # 撤销竞标

# === 消息相关（新增）===
GET    /api/messages                   # 获取消息列表
PUT    /api/messages/:id/read          # 标记消息已读
PUT    /api/messages/read-all          # 标记所有消息已读
DELETE /api/messages/:id               # 删除消息
```

**重要**：上述 API 设计完全对齐现有服务层契约，不再定义冲突接口。

## 五、功能详细设计（修订版）

### 5.1 个人资料模块

#### 5.1.1 资料卡片展示

**显示内容**：
- 头像（默认头像或用户上传）
- 姓名
- 角色标签（客户/开发者）
- 简介（最多显示 3 行，超出截断）
- 实名认证状态徽章

**交互**：
- 点击「编辑资料」打开模态框
- 模态框包含：头像上传、姓名、简介、地区、网站等字段

```javascript
// 示例：关闭任务
async function closeTask(taskId, version) {
  // 1. 乐观更新 UI
  updateTaskInList(taskId, { status: 'closed', canClose: false });

  try {
    // 2. 调用 API，携带版本号
    await api.closeTask(taskId, { version });

    // 3. 成功，保持 UI 状态
  } catch (error) {
    if (error.code === 'VERSION_CONFLICT') {
      // 4. 版本冲突，刷新列表
      showMessage('任务已被他人修改，正在刷新...');
      await refreshTaskList();
    } else {
      // 5. 其他错误，回滚 UI
      rollbackTaskInList(taskId);
      showMessage('操作失败，请重试');
    }
  }
}
```

## 九、回滚方案（修订版，移除错误假设）

### 9.1 代码回滚（修订版）

**原则**：不使用 force push，采用 revert 或 rollback branch。

```bash
# 方案 1：Revert（推荐）
# 1. 创建 revert 提交
git revert <commit-hash>

# 2. 如果有多个提交，使用范围
git revert <old-commit-hash>..<new-commit-hash>

# 方案 2：Rollback Branch（紧急情况）
# 1. 创建回滚分支
git checkout -b rollback/<version>
git checkout <stable-commit-hash>

# 2. 切换主分支到回滚分支
git checkout master
git merge rollback/<version>

# 方案 3：Reset（仅本地开发，禁止 push）
git reset --hard <stable-commit-hash>
# 警告：永远不要 push --force 到 master
```

### 9.2 数据回滚（修订版）

**Mock 数据**：直接删除新增的 Mock 文件

**用户数据（LocalStorage）**：
- 清除浏览器缓存会丢失数据（这是预期行为）
- 不提供"自动恢复"机制（之前的设计错误）
- 如果需要数据迁移，提供版本号和迁移脚本

**数据迁移方案（新增）**：
```javascript

const DATA_VERSION = '2.0';

// 检查数据版本
const storedVersion = localStorage.getItem('userDataVersion');
if (storedVersion !== DATA_VERSION) {
  migrateUserData(storedVersion, DATA_VERSION);
}

function migrateUserData(from, to) {
  // 迁移逻辑
  localStorage.setItem('userDataVersion', to);
}
```
